# WBS作成のコツ

WBS（Work Breakdown Structure：作業分解構成図）は、プロジェクトを細かいタスクに分解して管理するための手法です。

---

## WBSとは

```
プロジェクト全体
    │
    ├── 大きな機能A
    │   ├── 中タスク1
    │   │   ├── 小タスク1-1
    │   │   └── 小タスク1-2
    │   └── 中タスク2
    │
    └── 大きな機能B
        ├── 中タスク3
        └── 中タスク4
```

**ポイント**：大きな作業を小さく分解して、管理しやすくする

---

## WBSを作る目的

| 目的 | 説明 |
|------|------|
| 作業の見える化 | 何をやるべきか全員が把握できる |
| 漏れ防止 | タスクの抜け漏れを防ぐ |
| 担当の明確化 | 誰が何をやるか決められる |
| 進捗管理 | 今どこまで終わったかわかる |
| 見積もり | 工数や期間を見積もれる |

---

## WBS作成の手順

### Step 1: ゴールを明確にする

**何を作るのか？** を最初に決める

```
例: レシピ共有アプリを作成する
- ユーザーがレシピを投稿できる
- レシピを検索できる
- お気に入り登録できる
```

### Step 2: 大きな機能に分解する

プロジェクトを主要な機能やフェーズに分ける

```
レシピアプリ
├── 1. 環境構築
├── 2. ユーザー機能
├── 3. レシピ機能
├── 4. 検索機能
├── 5. お気に入り機能
└── 6. テスト・デバッグ
```

### Step 3: 中タスクに分解する

各機能をさらに具体的な作業に分ける

```
2. ユーザー機能
├── 2.1 データベース設計
├── 2.2 ユーザー登録
├── 2.3 ログイン
├── 2.4 ログアウト
└── 2.5 プロフィール表示
```

### Step 4: 小タスクに分解する

実際に作業できる単位まで分解する

```
2.2 ユーザー登録
├── 2.2.2 UserDTO作成
├── 2.2.3 UserDAO作成（registerメソッド）
├── 2.2.4 RegisterServlet作成
├── 2.2.5 register.jsp作成
├── 2.2.6 CSS適用
└── 2.2.7 動作確認
```

### Step 5: 担当者と期限を設定

各タスクに担当者と期限を割り当てる

| No | タスク | 担当 | 期限 | 状態 |
|----|--------|------|------|------|
| 2.2.1 | usersテーブル作成 | 田中 | 1/15 | 完了 |
| 2.2.2 | UserDTO作成 | 田中 | 1/15 | 完了 |
| 2.2.3 | UserDAO作成 | 山田 | 1/16 | 作業中 |
| 2.2.4 | RegisterServlet作成 | 山田 | 1/17 | 未着手 |

## 分解の目安

### どこまで分解するか？

**1〜2日で終わる作業** になるまで分解する

```
NG（大きすぎる）:
└── ユーザー機能を作る（1週間）

OK（適切なサイズ）:
├── UserDTO作成（30分）
├── UserDAO作成（2時間）
├── RegisterServlet作成（2時間）
└── register.jsp作成（1時間）
```

### 分解の基準

| 基準 | 説明 |
|------|------|
| 1人で完結 | 他の人を待たなくてもできる |
| 成果物が明確 | 何ができたら完了かわかる |
| 見積もり可能 | 何時間かかるか予想できる |
| 1〜2日以内 | 長くても2日で終わる |

## WBS作成のコツ

### コツ1: 動詞で終わる

```
NG: UserDTO
OK: UserDTOを作成する

NG: ログイン画面
OK: login.jspを作成する
```

### コツ2: 成果物を明確にする

```
NG: データベースを考える
OK: usersテーブルのCREATE文を作成する

NG: デザインを検討する
OK: トップページのワイヤーフレームを作成する
```

### コツ3: 依存関係を意識する

```
順番が重要なタスク:
1. テーブル作成 → 2. DTO作成 → 3. DAO作成 → 4. Servlet作成 → 5. JSP作成

並行して進められるタスク:
- Aさん: UserDAO作成
- Bさん: RecipeDAO作成（別テーブルなので並行可能）
```

### コツ4: 100%ルール

親タスクを全ての子タスクで完全にカバーする

```
NG（漏れがある）:
ユーザー機能
├── ログイン
└── ログアウト
   （登録が抜けている！）

OK（100%カバー）:
ユーザー機能
├── ユーザー登録
├── ログイン
├── ログアウト
└── プロフィール
```

### コツ5: 8/80ルール

- 最小: 8時間（1日）以上
- 最大: 80時間（2週間）以下

```
NG（小さすぎる）:
└── 変数名を決める（10分）

NG（大きすぎる）:
└── アプリを完成させる（1ヶ月）

OK（適切）:
└── RecipeDAOを作成する（4時間）
```

## チーム開発でのWBS活用

### 担当割り当てのコツ

```
機能ごとに担当を分ける:
├── ユーザー機能 → Aさん
├── レシピ機能 → Bさん
├── 検索機能 → Cさん
└── お気に入り機能 → Dさん

レイヤーごとに担当を分ける:
├── DB設計・DAO → Aさん（バックエンド得意）
├── Servlet → Bさん
└── JSP・CSS → Cさん（フロントエンド得意）
```

### 進捗管理の方法

**ステータスの例**:

| ステータス | 意味 |
|-----------|------|
| 未着手 | まだ始めていない |
| 作業中 | 今やっている |
| レビュー待ち | 完了したがチェック待ち |
| 完了 | 終わった |
| 保留 | 何らかの理由で中断 |

**進捗確認のタイミング**:
- 毎日の朝会（5〜10分）
- 週1回の振り返り

## WBS作成チェックリスト

作成後に確認：

- [ ] ゴールが明確か
- [ ] タスクに漏れがないか（100%ルール）
- [ ] 各タスクが1〜2日で終わるサイズか
- [ ] 成果物が明確か
- [ ] 依存関係を考慮しているか
- [ ] 担当者が決まっているか
- [ ] 期限が設定されているか

## まとめ

### WBS作成の流れ

```
1. ゴールを決める
      ↓
2. 大きな機能に分ける
      ↓
3. 中タスクに分ける
      ↓
4. 小タスク（1〜2日）に分ける
      ↓
5. 担当と期限を決める
      ↓
6. 進捗を管理する
```

### 覚えておくこと

1. **1〜2日で終わるサイズ** に分解する
2. **成果物を明確** にする（動詞で書く）
3. **依存関係** を意識する
4. **100%ルール** で漏れをなくす
5. **定期的に進捗確認** する
